## Alternatives Considered

### Alternative: External partner API for real-time lender matching
**Strengths:** Access to a large network of lenders without manual onboarding, real-time availability data, lender routing managed by the network
**Rejected because:**

WRITE THIS YOURSELF
Key points:
- External partner APIs (Optimal Blue, LenderPrice, lead networks) have integration costs,
  API rate limits, and availability dependencies. A lead routing failure because an external
  API is down is unacceptable for a paid lead.
- At Phase 1-2 scale, the number of contracted lenders is small (5-20 partners).
  A local database table is the correct data store for a known, bounded set of records.
- External APIs are appropriate when the lender network must scale beyond directly contracted
  partners — i.e., when the platform becomes a marketplace. That is not the Phase 1-2 scope.
- Lead routing logic (filter by state license, loan type, active status) is a simple SQL query
  against the local directory. No external API call needed at routing time.

### Alternative: Hardcoded lender configuration (YAML/properties file)
**Strengths:** Zero database dependency for lender configuration, simple to inspect and version-control
**Rejected because:**

WRITE THIS YOURSELF
Key points:
- Lender webhook endpoints, contact information, and licensed states change over time.
  Updating a YAML file requires a code deployment for every lender configuration change.
- The admin UI (ADR-0045) provides a proper lender management interface — the database
  is the correct backing store for admin-managed configuration data.
- A properties file approach is appropriate for a single fixed integration. A directory of
  multiple lenders with routing logic is relational data and belongs in a database.

## Rationale

### Local Database Ownership Is Correct for a Contracted Lead Network

WRITE THIS YOURSELF
A lead aggregator has a contracted relationship with each lender — they have an agreement,
an agreed-upon lead price, delivery format (webhook), and licensed states. This is structured
relational data that belongs in the system's database.

The lender table is simple:
  lenders (id, name, nmls_id, webhook_url, webhook_secret, active, created_at)
  lender_states (lender_id, state_code)  -- states where lender is licensed
  lender_loan_types (lender_id, loan_type)  -- FHA, VA, CONVENTIONAL, etc.

Lead routing is a SQL query:
  SELECT l.* FROM lenders l
  JOIN lender_states ls ON l.id = ls.lender_id AND ls.state_code = :borrowerState
  JOIN lender_loan_types llt ON l.id = llt.lender_id AND llt.loan_type = :loanType
  WHERE l.active = TRUE

This is correct, fast, and has no external dependencies. The local database is the
right source of truth for the system's contracted lender network.
